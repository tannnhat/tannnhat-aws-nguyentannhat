## Deploy Frontend Stack

Build the React SPA and deploy it to S3 + CloudFront, then configure runtime config so the browser reads the correct API URL and Cognito settings without them being baked into the JavaScript bundle.

---

### 5.3.1 Build the Frontend

```bash
cd ../frontend
npm run build
```

Expected:
```
vite v7.x.x building client environment for production...
✓ 2504 modules transformed.
dist/index.html                         1.14 kB │ gzip:   0.59 kB
dist/assets/index-XXXXXXXX.js         750.06 kB │ gzip: 217.53 kB
✓ built in 35s
```

> **Note:** The build may print a chunk size warning (`Some chunks are larger than 500 kB`). This is a performance suggestion, not an error — the build has succeeded when you see `✓ built in Xs`.

The compiled output is in `frontend/dist/`.

---

### 5.3.2 Deploy FrontendStack

**Prerequisites:**
- `WAF_WEB_ACL_ARN` is set in `infrastructure/.env` (from SecurityStack — step 5.2.5)
- `frontend/dist/` exists (built in step 5.3.1)

```bash
cd ../infrastructure
npx cdk deploy FrontendStack --require-approval never
```

This step takes **5–10 minutes** — CDK uploads all assets to S3 and creates the CloudFront distribution.

Expected:
```
FrontendStack

Outputs:
FrontendStack.CloudFrontDistributionId = EXXXXXXXXXXXXXXXXX
FrontendStack.CloudFrontDomainName = dXXXXXXXXXXXXXX.cloudfront.net
FrontendStack.FrontendBucketName = ecommerce-frontend-123456789012-ap-southeast-1
FrontendStack.FrontendUrl = https://dXXXXXXXXXXXXXX.cloudfront.net
```
---

### 5.3.3 Configure Runtime Config

The frontend reads API URL and Cognito config from `/config.json` served by CloudFront. This file is generated by CDK at deploy time — no values are hardcoded in the JavaScript bundle.

**Verify config.json is correct:**

> On **Linux/macOS** use `python3`. On **Windows** use `python`.

```bash
curl -s "https://dXXXXXXXXXXXXXX.cloudfront.net/config.json" | python3 -m json.tool
```

Expected:
```json
{
    "apiUrl": "https://xxxxxxxxxx.execute-api.ap-southeast-1.amazonaws.com/prod/",
    "cognito": {
        "userPoolId": "ap-southeast-1_XXXXXXXXX",
        "clientId": "xxxxxxxxxxxxxxxxxxxxxxxxxx",
        "region": "ap-southeast-1"
    }
}
```

---

### 5.3.4 Update CORS and Redeploy

Now that you have the CloudFront URL, update the API's CORS origin and redeploy.

**Step 1 — Update `infrastructure/.env`:**
```bash
ALLOWED_ORIGIN=https://dXXXXXXXXXXXXXX.cloudfront.net
```
> Use the actual `FrontendStack.FrontendUrl` value from the outputs above for the CloudFront URL.

**Step 2 — Redeploy ApiStack** to apply the new CORS origin to all three Lambda functions:
```bash
npx cdk deploy ApiStack --require-approval never
```

Expected — Lambda functions updated:
```
ApiStack |  UPDATE_COMPLETE | AWS::Lambda::Function | ProductServiceFunction
ApiStack |  UPDATE_COMPLETE | AWS::Lambda::Function | CartServiceFunction
ApiStack |  UPDATE_COMPLETE | AWS::Lambda::Function | OrderServiceFunction
ApiStack
```

**Step 3 — Rebuild frontend and redeploy FrontendStack** to regenerate `config.json` with the correct values:
```bash
cd ../frontend && npm run build && cd ../infrastructure
npx cdk deploy FrontendStack --require-approval never
```

### 5.3.5 Deploy InfrastructureStack

```bash
npx cdk deploy InfrastructureStack --require-approval never
```

Expected:
```
InfrastructureStack (no changes)
```

### Troubleshooting

| Problem | Solution |
|---------|---------|
| `dist/` folder not found | Run `npm run build` in `frontend/` first |
| CloudFront returns 403 | Wait 2–3 minutes for the distribution to propagate |
| `config.json` shows wrong values | Invalidate CloudFront cache: `aws cloudfront create-invalidation --distribution-id EXXXXXXXXX --paths "/*"` |
| CORS error in browser console | `ALLOWED_ORIGIN` not set correctly; redo step 5.3.4 |
| `WAF_WEB_ACL_ARN` is empty warning | Copy ARN from SecurityStack outputs to `infrastructure/.env` |
